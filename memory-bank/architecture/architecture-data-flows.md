## 1. Pipeline Overview

```
Plan → [Plan Regen] → Generate → Review → [Content Regen] → Deploy → Post → Analyze → (feeds next Plan)
```

| Step | Trigger | Script(s) | Output |
|------|---------|-----------|--------|
| 1. Weekly Plan | Monday 6am ET cron | `generate_weekly_plan.py` | `data/weekly-plan-W{N}-{date}.json`, Weekly Review sheet |
| 1b. Plan Regen (optional) | Sheet B5 = "regen" | `regen_weekly_plan.py` | Updated plan JSON + Weekly Review sheet |
| 2. Plan Approval | Sheet B3 = "approved" | Apps Script -> `generate-content.yml` | -- |
| 3a. Blog Generation | Workflow | `generate_blog_posts.py` -> `blog_generator.py` | `data/generated/blog/{slug}.mdx`, `data/blog-generation-results.json` |
| 3b. Pin Generation | Workflow | `generate_pin_content.py` | `data/generated/pins/`, `data/pin-generation-results.json` |
| 3c. Publish Queue | Workflow | `publish_content_queue.py` | GCS URLs, Content Queue sheet |
| 4. Content Review | Human in Sheet | -- | Status per row in Column J |
| 4b. Content Regen (optional) | Sheet N1 = "run" (via button or manual) | `regen_content.py` | Updated pins/images/sheet rows |

| 10. Monthly Review | 1st Monday | `monthly_review.py` | Strategy recommendations |

## 2. ID & Naming Conventions

### Post IDs
- **Format:** `W{week_number}-P{seq}` (e.g., `W12-P01`, `W12-P08`)
- **Generated by:** Claude in `generate_weekly_plan()` response
- **Unique within:** A single week
- **Used in:** weekly plan JSON -> blog-generation-results.json -> Content Queue sheet -> content-log.jsonl

### Pin IDs
- **Format:** `W{week_number}-{seq}` (e.g., `W12-01`, `W12-28`)
- **Generated by:** Claude in `generate_weekly_plan()` response
- **Unique within:** A single week
- **Used in:** weekly plan JSON -> pin-generation-results.json -> pin-schedule.json -> Content Queue sheet -> content-log.jsonl

### Pin-to-Blog Linkage
- **Field:** `source_post_id` on pin specs -> parent blog's `post_id`
- **Example:** Pin `W12-01` has `source_post_id: "W12-P01"`
- **Used for:** Deriving blog URL for pin link, mapping hero images between blog/pin, grouping pins under their parent blog

### Blog Slugs
- **Format:** lowercase-hyphenated (e.g., `easy-weeknight-chicken-tacos`)
- **Generated by:** Claude in blog post frontmatter `slug:` field
- **CRITICAL:** goslated.com routes by FILENAME, not frontmatter slug. The filename IS the slug.
- **Used for:**
  - Blog file: `data/generated/blog/{slug}.mdx`
  - Blog URL: `https://goslated.com/blog/{slug}`
  - Blog hero image: `data/generated/pins/{slug}-hero.{ext}`
  - Deployed blog: `content/blog/{slug}.mdx` on goslated.com repo
  - Deployed hero: `public/assets/blog/{slug}{ext}` on goslated.com repo

### Image Filenames
| Type | Pattern | Example | Where Saved |
|------|---------|---------|-------------|
| Pin hero image | `{pin_id}-hero.{ext}` | `W12-01-hero.png` | `data/generated/pins/` |
| Blog hero image (copy) | `{slug}-hero.{ext}` | `easy-chicken-tacos-hero.png` | `data/generated/pins/` |
| Rendered pin | `{pin_id}.png` | `W12-01.png` | `data/generated/pins/` |
| Regen hero (downloaded) | `{pin_id}-hero-downloaded.png` | `W12-01-hero-downloaded.png` | `data/generated/pins/` |

> **Removed in simplification:** `{pin_id}-ai-hero.png` (AI comparison hero) no longer generated. Stock images no longer sourced, so no separate "AI alternative" is needed.

### Image IDs (for dedup tracking)
| Source | Format | Example |
|--------|--------|---------|
| AI generated | `ai_{md5_hash[:12]}` | `ai_a1b2c3d4e5f6` |
| Template-only | `""` (empty string) | |

> **Removed in simplification:** `unsplash:{photo_id}` and `pexels:{photo_id}` formats no longer used. All images are now AI-generated. Historical entries in content-log.jsonl may still contain these prefixes.

- **Stored in:** `content-log.jsonl` and `pin-generation-results.json` -> `image_id` field
- **Dedup window:** 90 days (via `load_used_image_ids()` in `generate_pin_content.py`)

## 3. File Path Conventions

### Generated Content (data/generated/)
```
data/generated/
+-- blog/
|   +-- easy-weeknight-chicken-tacos.mdx     # Blog post (slug-named)
|   +-- 7-slow-cooker-dinners.mdx
|   +-- ...
+-- pins/
    +-- W12-01-hero.png          # Pin hero image (pin_id-named, AI-generated)
    +-- W12-01.png               # Rendered pin (1000x1500)
    +-- easy-weeknight-chicken-tacos-hero.png  # Blog hero image COPY (slug-named)
    +-- ...
```

### Dual Hero Image Save (generate_pin_content.py:177-183)

When a pin's hero image is generated, it's saved with the pin_id name (e.g., `W12-01-hero.png`). Then a COPY is made with the blog slug name (e.g., `easy-chicken-tacos-hero.png`). This is because:

- **Pin assembly** uses the pin_id-named file (`pin_assembler.assemble_pin(hero_image_path=...)`)
- **Blog deployment** looks for the slug-named file (`blog_deployer.py` -> commits `{slug}-hero.{ext}` to goslated.com)
- **GCS upload** uses the pin_id-named file for Content Queue previews

```python
# generate_pin_content.py lines 177-183
blog_slug = _resolve_blog_slug(pin_spec, blog_posts)
if blog_slug and image_path and Path(image_path).exists():
    slug_hero = PIN_OUTPUT_DIR / f"{blog_slug}-hero{Path(image_path).suffix}"
    if not slug_hero.exists():
        shutil.copy2(image_path, slug_hero)
```

### Data Files (data/)
```
data/
+-- weekly-plan-W12-2026-02-24.json    # Current week's plan
+-- pin-generation-results.json         # Pin generation output (updated by publish + regen)
+-- blog-generation-results.json        # Blog generation output
+-- pin-schedule.json                   # Posting schedule (created at promote step)
+-- content-log.jsonl                   # Append-only posting log (one JSON per line)
+-- token-store.json                    # Pinterest OAuth tokens (NEVER read)
+-- board-ids.json                      # Board name -> board ID mapping (optional)
```

### Deployed to goslated.com
```
goslated.com repo:
+-- content/blog/{slug}.mdx                    # Blog posts
+-- public/assets/blog/{slug}-hero.{ext}       # Hero images
```

## 4. Image Flow (Detailed)

### Current Architecture (Post-Simplification)

All images are AI-generated via `gpt-image-1.5`. No stock photo APIs, no Claude vision ranking/validation, no AI comparison images.

```
generate_pin_content()
  |
  +- Check pin_template
  |  +- "infographic-pin" -> template-only (no image needed)
  |  +- all others -> _source_ai_image()
  |
  +- _source_ai_image()
  |  +- claude.generate_image_prompt(pin_spec)  [GPT-5 Mini primary, Claude Sonnet fallback]
  |  +- Strip markdown code fences from GPT-5 Mini response if present
  |  +- Parse JSON to extract image_prompt field
  |  +- Append _regen_feedback if present (from content regen flow)
  |  +- image_gen_api.generate(prompt, 1000, 1500, style="natural")  [gpt-image-1.5]
  |  +- Hash prompt -> image_id = "ai_{md5[:12]}"
  |  +- Return (path, "ai_generated", image_id, quality_meta)
  |
  +- Dual save: {pin_id}-hero.png + {slug}-hero.png
  |
  +- pin_assembler.assemble_pin(hero_image_path=...)
     +- Puppeteer renders HTML -> PNG (1000x1500)
```

### Key Functions and Return Types

**`_source_ai_image()` returns:**
```python
(
    image_path: Path,           # e.g., data/generated/pins/W12-01-hero.png
    image_source: str,          # "ai_generated"
    image_id: str,              # "ai_a1b2c3d4e5f6"
    quality_meta: dict          # {"image_retries": 0}
)
```

Human review in the Content Queue is now the sole quality gate for images. There is no automated scoring/validation.

### Removed Image Flow (Historical Reference)

The pre-simplification pipeline had a multi-tier image system. This is documented here for historical context and to understand legacy data in content-log.jsonl.

```
[REMOVED] Pre-simplification flow:
  +- source_image() -- router by image_source_tier
  |  +- "stock" -> _source_stock_image()
  |  |   +- claude.generate_image_prompt(image_source="stock") -> search queries
  |  |   +- ImageStockAPI.search() -> Unsplash/Pexels API calls
  |  |   +- ImageStockAPI.filter_previously_used() -> dedup
  |  |   +- claude.rank_stock_candidates() [Haiku + vision] -> quality score
  |  |   +- If score < 6.5: retry broader query
  |  |   +- If retry < 5.0: fall back to AI generation
  |  |   +- ImageStockAPI.download_image()
  |  |
  |  +- "ai" -> _source_ai_image()
  |  |   +- claude.generate_image_prompt(image_source="ai")
  |  |   +- image_gen_api.generate() [gpt-image-1.5]
  |  |   +- claude.validate_ai_image() [Sonnet + vision] -> quality score
  |  |   +- If score < 6.5: regen with feedback, re-validate
  |  |   +- If still < 6.5: accept with low_confidence=True
  |  |
  |  +- "template" -> (None, "template", "", {})
  |
  +- [If winner is NOT AI] Generate AI comparison image
     +- _source_ai_image(filename_prefix="{pin_id}-ai")
        +- Stored as {pin_id}-ai-hero.png for Column M display

Deleted files: src/apis/image_stock.py (ImageStockAPI class)
Deleted functions: rank_stock_candidates(), validate_ai_image(), _process_ai_image_swaps()
Deleted prompts: image_rank.md, image_validate.md
```

## 5. Pin Assembly & Template Variables

### Template Types and Required Variables

| Template | Image Variable | Other Required | No Image |
|----------|---------------|----------------|----------|
| recipe-pin | `hero_image_url` | `headline`, `subtitle` | |
| tip-pin | `background_image_url` | `headline`, `bullet_1`, `bullet_2`, `bullet_3` | |
| listicle-pin | `background_image_url` | `number`, `headline`, `list_items` | |
| problem-solution-pin | `background_image_url` | `problem_text`, `solution_text` | |
| infographic-pin | -- | `title`, `steps`, `footer_text` | YES |

### How Image Variables Are Set

**In `assemble_pin()` (pin_assembler.py):**
```python
context = {"headline": headline, "subtitle": subtitle}
if hero_image_path:
    context["hero_image_url"] = str(hero_image_path)    # For recipe-pin
if extra_context:
    context.update(extra_context)                         # Includes background_image_url
```

**In `build_template_context()` (generate_pin_content.py:482-549):**
```python
# Non-recipe templates need background_image_url
if template_type != "recipe-pin" and image_path:
    context["background_image_url"] = str(image_path)
```

**At render time (`_inject_variables()`, pin_assembler.py):**
Both `hero_image_url` and `background_image_url` are converted from local file paths to data URIs:
```python
image_vars = ["hero_image_url", "background_image_url"]
for var in image_vars:
    if "{{" + var + "}}" in html:
        value = context.get(var, "")
        if value and not value.startswith(("data:", "http://", "https://")):
            value = _image_to_data_uri(value)  # Local path -> base64 data URI
```

### Call Site in generate_pin_content.py

```python
# Lines ~161-173
extra_context = build_template_context(
    template_type, pin_copy, pin_spec, image_path,
)
rendered_pin_path = assembler.assemble_pin(
    template_type=template_type,
    hero_image_path=image_path,
    headline=headline,
    subtitle=subtitle,
    variant=pin_spec.get("template_variant", 1),
    output_path=PIN_OUTPUT_DIR / f"{pin_id}.png",
    extra_context=extra_context,
)
```

## 8. GCS Upload Flow

### Sequence (publish_content_queue.py)

```
1. gcs.upload_pin_images(generated_pins, PIN_OUTPUT_DIR)
   -> Cleans up old images: delete_old_week_images(current_week)
      keeps W(current) and W(current-1), deletes W(current-2) and earlier
      also cleans ai-heroes/W* objects with same week logic
   -> Uploads rendered pins: {pin_id}.png
   -> Returns: {pin_id: gcs_public_url}
   -> Stored as pin["_drive_image_url"] and pin["_drive_download_url"]

2. gcs.upload_blog_hero_images(blog_results, PIN_OUTPUT_DIR)
   -> Uploads blog heroes: {slug}-hero.{ext}
   -> Returns: {post_id: gcs_public_url}

3. Save updated pin-generation-results.json (now with GCS URLs)

4. sheets.write_content_queue(
     blogs, pins,
     pin_image_urls,         # For Column I thumbnails
     blog_image_urls,        # For blog row thumbnails
     blog_previews,          # Frontmatter descriptions for blog rows
     quality_gate_stats,     # Summary row at bottom
   )

5. Write regen trigger cells: M1:N1 = ["Regen >", "idle"]
```

> **Removed in simplification:** `gcs.upload_ai_hero_images()` step removed. No AI comparison heroes are generated or uploaded. The `ai_image_urls` and `blog_ai_image_urls` parameters were removed from `write_content_queue()`.

### GCS URL Format
- Rendered pins: `https://storage.googleapis.com/{bucket}/W12-01.png`
- Blog heroes: `https://storage.googleapis.com/{bucket}/blog-heroes/easy-chicken-tacos-hero.png`

### Drive Fallback

If GCS upload fails, `publish_content_queue.py` falls back to Google Drive upload via `DriveAPI`. The same image URLs are stored but in Drive format. Regen also uses this fallback chain (GCS first, Drive if GCS fails).

## Appendix A: Plan-Level Regen Flow

Comprehensive documentation of `regen_weekly_plan.py` -- the plan-level topic replacement system.

### A. Trigger Chain

```
Reviewer sets Status="regen" + Feedback in Weekly Review blog post rows (Col F/G)
  -> Reviewer clicks "Run Plan Regen" button (or manually sets B5 = "regen")
  -> Apps Script onSheetEdit() detects B5 = "regen"
  -> Apps Script dispatches repository_dispatch: regen-plan
  -> GitHub Actions regen-plan.yml runs regen_weekly_plan.py
  -> regen_weekly_plan.py reads blog rows via sheets.read_plan_regen_requests()
```

**Timing:** Designed for the review phase BEFORE content generation (before B3 = "approved"). If triggered after content generation, the updated plan won't affect already-generated content.

### B. Full `regen_plan()` Orchestration (10-step flow)

```
regen_plan()
  |
  +- Step 1: Read regen requests from Weekly Review sheet
  |           -> sheets.read_plan_regen_requests()
  |           -> Returns list of {post_id, content_type, topic, pillar, keywords, status, feedback, row_index}
  |           -> If empty -> reset B5 trigger and return
  |
  +- Step 2: Load current weekly plan JSON from disk
  |           -> find_latest_plan() -> sorted glob of data/weekly-plan-*.json
  |
  +- Step 3: Build violations from regen requests
  |           -> build_regen_violations() converts sheet rows to violation dicts
  |           -> identify_replaceable_posts() (from generate_weekly_plan.py) finds posts + derived pins
  |
  +- Step 4: Build context for Claude
  |   +- posts_to_replace with violations attached
  |   +- all_slots (pin slots from affected posts)
  |   +- plan_context: kept topics, board counts, pillar counts
  |   +- reviewer_feedback dict: {post_id: feedback_string}
  |   +- content_memory from load_content_memory()
  |
  +- Step 5: Call Claude to generate replacements
  |           -> claude.generate_replacement_posts(
  |                posts_to_replace, slots_to_fill, plan_context,
  |                content_memory, negative_keywords=[],
  |                recent_topics=[], reviewer_feedback=reviewer_feedback
  |              )
  |           -> Uses weekly_plan_replace.md prompt template
  |
  +- Step 6: Splice replacements into the plan
  |           -> splice_replacements() from generate_weekly_plan.py
  |           -> Replaces blog posts and pins by ID while preserving the rest
  |
  +- Step 7: Save updated plan JSON (overwrites same file)
  |
  +- Step 8: Re-write Weekly Review sheet with updated plan
  |           -> Preserves B3 (plan approval) and B4 (deploy status) values
  |
  +- Step 9: Reset B5 trigger to "idle"
  |
  +- Step 10: Send Slack notification with replacement summary
```

### C. Key Constraints
- Plan regen does NOT trigger content generation. It only replaces topics/pins in the plan JSON.
- After plan regen, the reviewer can approve (B3 = "approved") to start content generation with the updated plan.
- If plan regen is triggered after B3 approval, generated content for replaced posts will be orphaned (not harmful, just wasted compute).

## Appendix B: Content Regen Flow

Comprehensive documentation of `regen_content.py` based on the actual implementation.

### A. Trigger Chain

```
Reviewer sets Column J status to regen_image/regen_copy/regen (in Content Queue)
  -> Reviewer clicks "Run Regen" button (or manually sets N1 = "run")
  -> Apps Script onSheetEdit() detects N1 = "run" (col 14)
  -> Apps Script dispatches repository_dispatch: regen-content
  -> GitHub Actions regen-content.yml runs regen_content.py
  -> regen_content.py reads Column J statuses via sheets.read_regen_requests()
```

**Note:** The Apps Script `onSheetEdit()` trigger dispatches the workflow, but `regen_content.py` itself reads row statuses from Column J to determine which items to process. The N1 trigger cell is just the execution gate.

### B. Full `regen()` Orchestration (7-step flow)

```
regen()
  |
  +- Step 1: Init API clients (Sheets, Claude, ImageGen, PinAssembler, GCS, Drive, Slack)
  |           Load used_image_ids, brand_voice, keyword_targets
  |
  +- Step 2: sheets.read_regen_requests()
  |           -> Returns list of {id, status, feedback, type, row_index}
  |           -> Status starts with "regen" (regen_image, regen_copy, regen)
  |           -> If empty -> reset trigger and return
  |
  +- Step 3: Load data files
  |   +- pin-generation-results.json -> pin_index {pin_id: pin_data}
  |   +- blog-generation-results.json -> blog_results {post_id: blog_data}
  |
  +- Step 4: Process each request
  |   +- Blog items -> _regen_blog_image() (copy regen N/A, redirected to image-only)
  |   +- Pin items -> _regen_item() (image, copy, or both)
  |       +- After: _update_pin_results() merges new data into pin_data in-place
  |       +- After: sheets.update_content_row() updates Sheet row
  |
  +- Step 5: Save updated files
  |   +- pin-generation-results.json (always)
  |   +- blog-generation-results.json (if any blog regens ran)
  |   +- pin-schedule.json (update 7 fields for affected pins, if file exists)
  |
  +- Step 6: sheets.reset_regen_trigger() -> sets N1 back to "idle"
  |
  +- Step 7: slack.notify_regen_complete(regen_results)
```

### C. Pin Regen: `_regen_item()`

Regenerates a single pin's image, copy, or both. All image regen is AI-only.

```
_regen_item(pin_data, regen_type, feedback, claude, image_gen_api, assembler, gcs, drive, ...)
  |
  +- Build pin_spec from pin_data (pin_template from template field)
  |   +- Attach _regen_feedback if feedback provided
  |
  +- [regen_copy / regen] Regenerate copy:
  |   +- generate_copy_batch(claude, [copy_spec], ...) with _copy_feedback
  |       -> Updates title, description, alt_text, text_overlay in new_pin_data
  |       -> On failure: sets _copy_regen_failed flag, preserves original copy
  |
  +- [regen_image / regen] Regenerate image:
  |   +- For infographic-pin: skip (template-only, no image)
  |   +- For all others: _source_ai_image(pin_spec, claude, image_gen_api, PIN_OUTPUT_DIR)
  |       -> Returns (image_path, "ai_generated", image_id, quality_meta)
  |       -> Update new_pin_data: hero_image_path, image_source, image_id, image_retries
  |
  +- Resolve hero image (download from GCS/Drive if not on local disk -- fresh runner)
  |
  +- Re-render pin:
  |   +- build_template_context() for template-specific vars
  |   +- assembler.assemble_pin() -> {pin_id}.png
  |   +- Upload rendered pin (GCS first, Drive fallback)
  |       -> Updates _drive_image_url, _drive_download_url in new_pin_data
  |
  +- Update Content Queue row:
      +- sheets.update_content_row(row_index, status="pending_review",
           thumbnail, title, description, notes, feedback="")
```

**Returns:** `{"pin_data": new_pin_data, "image_url": new_gcs_url}`

### D. Blog Image Regen: `_regen_blog_image()`

Blog copy regen is not supported (blog text lives in MDX files). Blog items with `regen` or `regen_copy` status are redirected to image-only regen.

```
_regen_blog_image(blog_data, post_id, feedback, claude, image_gen_api, gcs, ...)
  |
  +- Build pin_spec from blog_data (title as primary_keyword, default template "recipe-pin")
  |   +- Attach _regen_feedback if feedback provided
  |
  +- _source_ai_image(pin_spec, claude, image_gen_api, PIN_OUTPUT_DIR)
  |
  +- Save hero with slug-based filename: {slug}-hero.{ext}
  |   +- shutil.copy2() from pin_id-named path to slug-named path
  |   +- Clean up stale heroes with different extensions
  |
  +- Upload raw hero to GCS (timestamped: {post_id}-hero-{ts}.{ext})
  |   +- Timestamp busts Google Sheets =IMAGE() cache
  |
  +- Update blog-generation-results.json metadata:
      _hero_image_source, _hero_image_id, _hero_quality_score,
      _hero_regen_feedback, _hero_regen_timestamp, _hero_gcs_url
```

**Returns:** `{"image_url", "quality_score", "image_source", "image_id"}`

### E. Pin-Schedule.json Updates

When `pin-schedule.json` exists (post-promote), regen updates **7 fields** for affected pins:

| # | Field | Source in pin-generation-results.json |
|---|-------|--------------------------------------|
| 1 | `title` | `title` |
| 2 | `description` | `description` |
| 3 | `alt_text` | `alt_text` |
| 4 | `image_path` | `image_path` |
| 5 | `image_url` | `_drive_download_url` |
| 6 | `image_source` | `image_source` |
| 7 | `image_id` | `image_id` |

### F. Error Handling

| Scenario | Behavior |
|----------|----------|
| Pin data not found in results JSON | -> `pending_review` with error note, skip item |
| Blog data not found in results JSON | -> `pending_review` with error note, skip item |
| Blog copy regen requested | -> Not supported, status reset with warning note |
| Image sourcing returns None | -> Error logged, hero_image_path not updated |
| Hero image not on disk (fresh CI runner) | -> Download from GCS (preferred) or Drive (fallback) |
| GCS upload fails | -> Fall back to Drive upload for rendered pin |
| Copy regen fails | -> `_copy_regen_failed` flag set, original copy preserved |
| Hero unavailable for re-render | -> `_copy_regen_no_rerender` flag, copy updated but pin image not re-rendered |
| Pin assembly/render fails | -> Error logged, image_url remains None |
| Sheet row update fails | -> Error logged, continues to next item |
| pin-schedule.json update fails | -> Error logged, schedule not updated (pins still work from previous data) |
