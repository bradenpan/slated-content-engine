## 1. High-Level Flow Overview

**File:** `src/generate_pin_content.py`

The main entry point is `generate_pin_content()` (line 47). For each pin in the weekly plan:

1. **Generate copy** via Claude Sonnet (`_generate_all_copy()`, line 106) — batches of 6 pins per API call using `prompts/pin_copy.md`
2. **Source image** based on `image_source_tier` from the plan (`_source_pin_image()`, line 133)
3. **Assemble final pin** — inject image + text into HTML/CSS template, render to 1000x1500 PNG via Puppeteer (`assembler.assemble_pin()`, line 165)
4. **Save results** to `data/pin-generation-results.json` (`_save_pin_results()`, line 241)

The image tier is determined by the `pin_spec["image_source_tier"]` field set during weekly plan generation. Valid values: `"stock"` / `"Tier 1"`, `"ai"` / `"Tier 2"`, `"template"` / `"Tier 3"`.

## 2. Entry Point and Dispatch

### `source_image()` — Line 297-352

**File:** `src/generate_pin_content.py:297-352`

This is the public dispatch function. It reads `image_tier` and routes to the appropriate handler:

```
image_tier == "stock" or "Tier 1"  -->  _source_stock_image()
image_tier == "ai" or "Tier 2"    -->  _source_ai_image()
image_tier == "template" or "Tier 3" --> returns (None, "template", "", template_meta)
unknown tier                        --> defaults to _source_stock_image() with warning
```

All paths return a 4-tuple: `(image_path, image_source, image_id, quality_meta)`.

The `quality_meta` dict tracks:
- `image_quality_score` (float or None)
- `image_retries` (int)
- `image_low_confidence` (bool)
- `image_source_original` (str — "stock", "ai", or "template")
- `image_quality_issues` (list[str])

### How it's called — Line 670-694

`_source_pin_image()` is a thin wrapper that calls `source_image()` with all the required arguments. Called from the main loop at line 133.

### Step 3.2: Search Stock APIs

The `search()` method:

3. Implements fallback: if the primary source returns nothing, tries the other

- Params: `query`, `per_page` (max 30), `orientation`
- Rate limit: 50 requests/hour

**Pexels API call** (`src/apis/image_stock.py:235-292`):
- Endpoint: `GET /v1/search`
- Auth: `Authorization: {key}` header
- Params: `query`, `per_page` (max 80), `orientation`
- Rate limit: 200 requests/hour

### Step 3.5: Rank Candidates with Claude Haiku Vision

**Line 783:**
```python

```

**File:** `src/apis/claude_api.py:673-772`

**Inside `rank_stock_candidates()`:**

- **Template:** Loads `prompts/image_rank.md`
- **Context variables (lines 700-705):**
  - `PIN_SPEC` = full pin_spec dict (serialized as JSON)
  - `PIN_TEMPLATE` = `pin_spec.get("pin_template", "recipe-pin")`
  - `NUM_CANDIDATES` = number of candidates
  - `MAX_INDEX` = number of candidates minus 1
- **System message (lines 718-722):**
  ```
  "You are a visual quality evaluator for Pinterest pin images.
  Evaluate each candidate image and return a JSON array of scores.
  IMPORTANT: Output ONLY valid JSON. No explanations or text before or after."
  ```
- **Model:** Claude Haiku (`MODEL_HAIKU = "claude-haiku-4-5-20251001"`)
- **Temperature:** 0.3
- **Max tokens:** 1024
- **Images:** Thumbnail bytes are sent as base64-encoded image content blocks

The response is parsed as a JSON array of `{index, score, reason}` objects. Scores are mapped back to the candidate dicts as `_score` and `_rank_reason` fields, then sorted descending.

**ISSUE IDENTIFIED:** During regen, `_regen_feedback` is buried inside the `PIN_SPEC` JSON dump. The `image_rank.md` prompt has no mention of regen feedback, so Haiku scores purely on visual match, ignoring why the previous image was rejected.

**ISSUE IDENTIFIED:** Claude Haiku misidentifies food at thumbnail resolution. At ~200px, food subjects can look similar. Example: scoring a bowl of lemons 8.0 for a "lemon herb chicken" pin.

The retry follows the same flow: search both APIs, filter duplicates AND previously-seen IDs, download thumbnails for top 5, rank with Haiku.

## 6. Alt Text Generation

Alt text is generated as part of the **pin copy generation** step, NOT as a separate image-analysis step.

### Where it happens

**File:** `src/generate_pin_content.py:106-112` — `_generate_all_copy()` calls `generate_copy_batch()`, which calls `claude.generate_pin_copy()`.

**File:** `src/apis/claude_api.py:196-272` — `generate_pin_copy()` uses `prompts/pin_copy.md`.

The prompt template includes these instructions for alt text:

```
### Alt Text (500 characters max)
- Describe what is physically IN the image. Be literal and visual.
- Include the primary keyword phrase naturally.
- Alt text is confirmed to improve impressions by ~25% (Tailwind study). Always fill it in.
- Format: "[Visual description of image]. [Primary keyword context]."
- Example: "Overhead view of a sheet pan with golden chicken thighs and roasted broccoli.
  Easy 30-minute weeknight dinner recipe for families."
```

The output JSON includes an `alt_text` field per pin.

### How alt text flows into the final pin data

**File:** `src/generate_pin_content.py:195` — stored as `pin_data["alt_text"]`

The alt text is part of the pin metadata but is NOT rendered into the pin image itself. It is used at posting time when the pin is published to Pinterest.

### Key observation

Alt text is generated BEFORE the image is sourced. It describes what the image SHOULD look like based on the pin topic, not what the image actually contains. This means the alt text might not accurately describe the actual selected image (e.g., if the stock image differs from the expected composition).

## 7. Pin Type Differences in Image Sourcing

### How image tier is determined

The `image_source_tier` is set during weekly plan generation (by the `weekly_plan.md` prompt), not during image sourcing. Different pin types have different default tiers:

- **recipe-pin:** Typically `"stock"` (Tier 1) — food photography is the primary subject
- **tip-pin:** `"stock"` or `"template"` — depends on whether a background image is needed
- **listicle-pin:** `"stock"` — typically a single hero dish image
- **problem-solution-pin:** `"template"` (Tier 3) — often works better without a photo
- **infographic-pin:** Always `"template"` (Tier 3) — uses styled backgrounds, no photography

### Template-specific search query adjustments

The `prompts/image_search.md` template has content-type-specific guidance:

- **Recipe Pins:** Focus on the finished dish. Overhead/flat-lay. Include cooking vessel. Specify key ingredients.
- **Tip/How-To Pins:** Ingredient flat-lays, organized kitchen scenes, meal prep layouts.
- **Listicle Pins:** Single hero dish or general food category image.
- **Problem-Solution Pins:** Lifestyle or abstract. Family at dinner table, kitchen scene. "Often better served by Tier 3."
- **Plan-Level Pins:** Ingredients grouped to represent a week of meals.

### Template-specific ranking adjustments

The `prompts/image_rank.md` has template-specific criteria:

- **recipe-pin:** Correct food subject, overhead/flat-lay, warm lighting, clear bottom third for text overlay
- **tip-pin:** Subject relevant to topic, clean enough for background behind text, good color contrast
- **listicle-pin:** Topical relevance, lifestyle quality, not too busy
- **problem-solution-pin:** Emotional resonance, clean composition for text overlay

### Template-specific AI prompt adjustments

The `prompts/image_prompt.md` has composition guidance by template:

- **Recipe Pin:** Finished dish, overhead/45-degree, 1-2 props, clear bottom third
- **Tip/How-To Pin:** Kitchen scene or ingredient arrangement, works as background with heavy text
- **Listicle Pin:** Single hero dish or collage arrangement
- **Problem-Solution Pin:** Lifestyle scene conveying the problem
- **Infographic Pin:** Explicitly says "Skip. No AI image needed."

### Template-specific validation adjustments

The `prompts/image_validate.md` has template-specific criteria:

- **recipe-pin:** Correct food subject, overhead composition, warm lighting, clear bottom 30-40%, realistic textures
- **tip-pin:** Subject relevant, clean background for text, good contrast
- **problem-solution-pin:** Emotional tone matches, not too literal

### Build template context — `build_template_context()` (Lines 543-610)

**File:** `src/generate_pin_content.py:543-610`

Non-recipe templates need additional variables beyond headline/subtitle:

- **tip-pin:** `bullet_1`, `bullet_2`, `bullet_3` — extracted from description sentences
- **listicle-pin:** `number`, `list_items` — extracted from headline and description
- **problem-solution-pin:** `problem_text`, `solution_text` — derived from headline + sub_text
- **infographic-pin:** `title`, `steps`, `footer_text` — extracted from description

For non-recipe templates, `image_path` is also set as `background_image_url` (line 570-571).

## 8. Pin Assembly (Rendering)

**File:** `src/pin_assembler.py`

### `assemble_pin()` — Lines 477-508

Maps the caller's arguments to a `context` dict and calls `render_pin()`:
- `headline` and `subtitle` from pin copy
- `hero_image_url` from the sourced image path
- `extra_context` merged in (template-specific variables)

### `render_pin()` — Lines 403-475

1. Loads the template HTML (`templates/pins/{template_name}/template.html`)
2. Loads base CSS (`templates/pins/shared/base-styles.css`) and template CSS (`templates/pins/{template_name}/styles.css`)
3. Inlines CSS into HTML
4. Activates the selected variant (A, B, or C) — hides other variants
5. Injects variables into HTML placeholders (`{{headline}}`, `{{hero_image_url}}`, etc.)
6. Converts local image paths to base64 data URIs for headless rendering
7. Writes final HTML to a temp file
8. Calls `render_pin.js` (Node.js/Puppeteer) to render to 1000x1500 PNG
9. Optimizes file size (target: <500KB)

### Image optimization — `_optimize_image()` (Lines 510-559)

If PNG > 500KB:
1. Re-save with PNG optimization
2. If still too large, convert to high-quality JPEG (quality=88)
3. Rename JPEG back to original .png path so downstream references remain valid

## 9. Full Prompt Templates

### 9.1. `prompts/image_search.md` — Stock Photo Search Query Generator

```markdown
# Stock Photo Search Query Generator

---
## SYSTEM

You are a visual content sourcer for Slated, a family meal planning app. You generate optimized search queries for stock photo APIs (Unsplash and Pexels) to find high-quality food and lifestyle photography for Pinterest pins and blog hero images.

You understand that stock photo search works differently from web search. Queries should be descriptive and visual, not conversational. You know what makes a great Pinterest food image: bright, overhead, warm-toned, clean composition.

---
## CONTEXT

### Pin Specification
- **Pin Topic:** {{pin_topic}}
- **Content Type:** {{content_type}}
- **Primary Keyword:** {{primary_keyword}}

---
## YOUR TASK

Generate 3-5 search queries optimized for Unsplash and Pexels APIs. These queries will be sent to both APIs, and the best result will be selected (by the pipeline script or by Claude evaluating thumbnails).

---
## OUTPUT FORMAT

Return valid JSON. No markdown code fences.

{
  "queries": [
    {
      "query": "the search query string",
      "api_hint": "unsplash | pexels | both",
      "rationale": "what this query targets"
    }
  ],
  "orientation": "portrait",
  "color_mood": "warm",
  "composition_preference": "overhead | angled | close-up | flat-lay | lifestyle",
  "subject_description": "Brief description of the ideal image for quality evaluation"
}

---
## QUERY WRITING RULES

### How Stock Photo Search Differs from Web Search
- **Shorter queries.** 2-5 words per query. Long phrases return worse results.
- **Descriptive, not conversational.** "chicken dinner plate overhead" not "what does a chicken dinner look like from above"
- **Visual language.** Describe what the image LOOKS like, not what it means.
- **Specific food terms work.** "chicken stir fry wok" returns relevant results. "easy weeknight dinner" returns nothing useful.
- **Include composition hints.** "overhead," "flat lay," "close up" are recognized by stock photo search.

### Query Strategy: Broad to Specific
Generate queries from most specific to most general:
1. **Specific dish query:** The exact food item with visual modifiers. "overhead chicken stir fry vegetables wok"
2. **Category query:** The food category with style modifiers. "asian chicken dinner plate styled"
3. **Ingredient-focused query:** Key ingredients as visual subjects. "chicken broccoli bell pepper cooking"
4. **Mood/scene query:** The setting or feeling. "family dinner table warm lighting" (use sparingly -- often too generic)
5. **Fallback query:** Simplified, broadest version. "chicken dinner" or "family meal"

### Content Type Adjustments

**Recipe Pins (most common):**
- Focus on the finished dish.
- Overhead / flat-lay compositions strongly preferred.
- Include the cooking vessel if relevant ("in a skillet," "on a sheet pan," "in a bowl").
- Specify key ingredients visually ("with vegetables," "with rice," "with pasta").

**Tip / How-To Pins:**
- May not need food photos at all (template-only designs).
- If a photo is needed: ingredient flat-lays, organized kitchen scenes, meal prep layouts.
- "meal prep containers organized" / "ingredients flat lay kitchen" / "cutting board vegetables organized"

**Listicle Pins:**
- Can use a single hero dish or a general food category image.
- "colorful dinner plates variety" / "multiple dishes table overhead"

**Problem-Solution Pins:**
- Lifestyle or abstract. Family at dinner table, kitchen scene, frustrated cooking (avoid faces).
- "kitchen counter messy cooking" / "dinner table setting warm" / "meal planning kitchen"
- Often better served by Tier 3 (template-only) -- only search stock if specifically assigned.

**Plan-Level Pins:**
- Ingredients grouped to represent a week of meals.
- "grocery ingredients organized groups" / "weekly meal ingredients flat lay" / "five dinner ingredients countertop"

### Unsplash vs. Pexels Differences

**Unsplash:**
- Higher quality individual photos, more editorial/artistic.
- Better for styled food photography and lifestyle shots.
- Search works well with short, descriptive phrases.
- Rate limit: 50 requests/hour.

**Pexels:**
- Larger volume of results, more variety in styles.
- Good for specific food items and general food photography.
- Search handles slightly longer queries better than Unsplash.
- Rate limit: 200 requests/hour.

**Set `api_hint` accordingly:**
- "unsplash" for editorial/styled food shots and lifestyle scenes.
- "pexels" for specific food items when you need more variety.
- "both" for most queries (search both, pick the best result).

---
## ORIENTATION AND STYLE PREFERENCES

- **Orientation:** Always "portrait" -- Pinterest pins are 2:3 ratio (1000x1500). Portrait images require less cropping.
- **Color mood:** Always "warm" -- warm-toned food photography with bright, natural lighting.
- **Avoid:** Dark/moody food photography, blue-toned images, heavily filtered images, images with text or watermarks.

## EXAMPLES

### Example 1: Chicken Stir Fry Recipe Pin
```

## 10. Known Issues and Failure Modes

### Issue 1: Only first search query is used (Stock path)

**Location:** `src/generate_pin_content.py:738`

Claude generates 3-5 search queries but only `queries[0]` is used. This limits the candidate pool and reduces the chance of finding a high-quality match. The `regen-quality-improvements.md` document proposes using multiple queries during regen.

### Issue 2: Regen feedback not surfaced to ranking model

**Location:** `src/apis/claude_api.py:718` (system message for `rank_stock_candidates()`)

During regen, `_regen_feedback` is embedded in the `PIN_SPEC` JSON dump but not highlighted. The `image_rank.md` prompt has no instructions about regen feedback. Haiku scores purely on visual match, ignoring why the previous image was rejected.

### Issue 3: Claude Haiku misidentifies food at thumbnail resolution

The ranking uses ~200px thumbnails with Claude Haiku (cheapest model). At small sizes, different food subjects can look similar. Example: scoring a bowl of citrus 8.0 for a "lemon herb chicken" pin.

### Issue 4: Alt text describes expected image, not actual image

Alt text is generated in the copy step before image sourcing. If the selected stock image differs from expectations (e.g., different plating style, different angle), the alt text may not accurately describe it.

### Issue 5: AI image validation failure path accepts low-quality images

After one retry, AI images that still score < 6.5 are accepted with only a `low_confidence` flag. There is no fallback to stock or template-only for AI images.

### Issue 6: `negative_prompt` field from `image_prompt.md` is never used

The `image_prompt.md` template asks Claude to generate a `negative_prompt` field, but the `_source_ai_image()` function only extracts `image_prompt` from the response. The `negative_prompt` is discarded. Neither the OpenAI gpt-image-1 nor Replicate Flux Pro APIs accept a `negative_prompt` parameter in the way Stable Diffusion does.

### Issue 7: No image search query/prompt is stored

Currently, the search query (stock) or AI prompt used to generate an image is never persisted to `pin-generation-results.json`. Only the `image_id` is saved. This makes debugging difficult. The `regen-quality-improvements.md` document proposes storing these.

### Issue 8: Stock search retry uses only the first query (same as initial)

The `stock_retry` path (line 803) generates a new query but again only uses the first result from Claude's response, same as the initial search.
