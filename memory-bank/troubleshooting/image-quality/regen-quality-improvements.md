# Regen Image Quality Improvements — Background & Plan

## How Image Regen Works (Full Trace)

When a pin row is marked `regen_image` with feedback, the flow is:

### Step 1: Build pin_spec and source a new image
`regen_content.py:302-317` builds a `pin_spec` dict from the existing pin_data. The feedback from column L is set as `pin_spec["_regen_feedback"]`.

`source_image()` is called (from `generate_pin_content.py`), which dispatches to `_source_stock_image()` or `_source_ai_image()` based on the original image source tier.

### Step 2: Generate search query (stock path)
`_source_stock_image()` at line 732 calls `claude.generate_image_prompt(pin_spec, image_source="stock", regen_feedback=feedback)`.

In `claude_api.py:380-428`:
- Loads `prompts/image_search.md` template
- Renders it with context: `pin_topic` (the pin title), `content_type`, `primary_keyword`, `pin_template`
- Sets system message: `"You are an image prompt specialist..."`
- **Appends feedback** to system message (line 414-418): `"IMPORTANT: The previous image was rejected by the reviewer with this feedback: {feedback}. Generate a search query that specifically addresses this feedback."`
- Calls Claude Sonnet at temperature 0.8
- Returns JSON with 3-5 search queries

**Problem: Only the first query is used.** Line 738 in `_source_stock_image()`: `search_query = parsed["queries"][0]["query"]`. The other 2-4 queries generated by Claude are discarded. During regen, casting a wider net matters — more queries = more candidate images to choose from.

### Step 3: Search stock APIs
`stock_api.search()` sends the single query to both Unsplash and Pexels, returning ~20 candidates total.

### Step 4: Rank candidates with Claude Haiku
Top 5 candidates get thumbnails downloaded. `claude.rank_stock_candidates(candidates, pin_spec)` sends the thumbnails to Claude Haiku for visual scoring.

The ranking uses `prompts/image_rank.md`:
- Criteria: subject relevance, composition, lighting, Pinterest-readiness
- Scoring: 9-10 perfect, 7-8 good, 5-6 acceptable, 3-4 poor, 1-2 unusable
- "Wrong food subject = score 1-2 regardless of photo quality"

**Problem: Regen feedback is buried in the PIN_SPEC JSON dump.** The `pin_spec` dict (including `_regen_feedback`) is rendered into the ranking prompt as `{{PIN_SPEC}}` — which becomes a raw JSON dump. The `_regen_feedback` field is in there but:
1. It's not highlighted or called out as an instruction
2. The `image_rank.md` prompt has no mention of regen feedback
3. Haiku has no instruction to "avoid images matching this complaint"
4. So Haiku scores purely on visual match to the topic, ignoring why the previous image was rejected

**Problem: Claude Haiku misidentifies food at thumbnail resolution.** In the W9-02 case (lemon herb chicken), Haiku gave 8.0 to an image of "lemon and vegetables in a bowl with no chicken." At small thumbnail sizes, food subjects can look similar and the cheaper model makes mistakes.

### Step 5: Select best, re-render, upload
If best score >= 6.5, that candidate is selected. If all < 6.5, a retry search with broader terms is attempted (using `stock_retry` mode). Below 5.0 on retry, it falls back to AI image generation.

## Change 2: Improve Regen Image Quality

Two targeted fixes:

### Fix 2A: Pass regen feedback to the ranking model

**File: `src/apis/claude_api.py`** (method `rank_stock_candidates`, around line 718)

The ranking model currently sees feedback only as a buried JSON field. Add it to the system message:

```python
# After line 722 (system = "You are a visual quality evaluator...")
regen_feedback = pin_spec.get("_regen_feedback", "")
if regen_feedback:
    system += (
        f" CRITICAL: This is a re-selection. The previous image was rejected with "
        f"this feedback: \"{regen_feedback}\". "
        f"Any candidate matching this complaint MUST score 1-2 regardless of photo quality."
    )
```

**Impact**: The ranking model will actively avoid the same mistake. For W9-02, if the feedback was "this is chicken with oranges, not lemon herb chicken," Haiku would score any orange-chicken images as 1-2.

### Fix 2B: Use multiple search queries during regen

**File: `src/generate_pin_content.py`** (method `_source_stock_image`, around line 735)

Currently only `queries[0]` is used. During regen, use multiple queries to expand the candidate pool:

```python
# Replace lines 735-742 with:
try:
    parsed = json.loads(search_query_raw)
    if isinstance(parsed, dict) and "queries" in parsed:
        all_queries = [q["query"] for q in parsed["queries"]]
    else:
        all_queries = [search_query_raw]
except (json.JSONDecodeError, KeyError, IndexError):
    all_queries = [search_query_raw]

search_query = all_queries[0]  # primary query (used for logging)

# During regen, search with multiple queries to expand the candidate pool
is_regen = bool(pin_spec.get("_regen_feedback"))
if is_regen and len(all_queries) > 1:
    all_candidates = []
    seen_ids = set()
    for q in all_queries[:3]:  # use up to 3 queries
        logger.info("Regen multi-query search for %s: '%s'", pin_id, q[:80])
        results = stock_api.search(query=q, num_results=10, orientation="portrait")
        for c in results:
            cid = c.get("id")
            if cid and cid not in seen_ids:
                seen_ids.add(cid)
                all_candidates.append(c)
    candidates = all_candidates
    logger.info("Regen multi-query produced %d unique candidates for %s", len(candidates), pin_id)
else:
    candidates = stock_api.search(query=search_query, num_results=10, orientation="portrait")
```

**Impact**: During regen, searches with up to 3 different queries (e.g., "overhead lemon herb chicken," "lemon chicken spring vegetables plate," "herb chicken one pan dinner"), producing ~30 unique candidates instead of ~20. More candidates = better chance of finding a quality match. Normal (non-regen) flow is unchanged.

### Files to modify
- `src/apis/claude_api.py` — `rank_stock_candidates()` system message
- `src/generate_pin_content.py` — `_source_stock_image()` multi-query logic

## Change 3: Store Image Prompts for Visibility

Currently, the search query (stock) or AI prompt used to generate an image is never stored. Only the image_id (Unsplash/Pexels ID or MD5 hash) is saved. This makes it impossible to:
- See what prompt produced a given image
- Compare prompts across regens
- Debug why a search returned poor results

### What to store

**File: `src/generate_pin_content.py`**

A. In `_source_stock_image()` (after line 743, where search_query is set):
```python
quality_meta["_image_search_query"] = search_query
```

B. In `_source_ai_image()` (after line 944, where image_prompt is constructed):
```python
quality_meta["_image_prompt"] = image_prompt
```

C. In pin_data construction (~line 215, add to the dict):
```python
"_image_prompt": quality_meta.get("_image_prompt"),
"_image_search_query": quality_meta.get("_image_search_query"),
```

**File: `src/regen_content.py`**

D. In `_update_pin_results()` (line 560, add to `keys_to_update` list):
```python
"_image_prompt", "_image_search_query"
```

E. In regen history entry (line 546, add to dict):
```python
"previous_image_prompt": pin_data.get("_image_prompt"),
"previous_search_query": pin_data.get("_image_search_query"),
```

### Result
After these changes, `pin-generation-results.json` will contain:
```json
{
  "pin_id": "W9-02",
  "_image_search_query": "overhead lemon herb chicken vegetables one pan",
  "_regen_history": [
    {
      "regen_type": "regen_image",

      "previous_search_query": "herb chicken spring vegetables"
    }
  ]
}
```

### Files to modify
- `src/generate_pin_content.py` — store prompt in quality_meta
