## Overall Assessment: PASS WITH NOTES

The research is thorough, the avoidance methods are well-sourced, and the implementation plan is well-structured with correct codebase references. The plan is actionable and would meaningfully reduce AI detection. However, there are several specific corrections, a missed pipeline entry point, and a couple of technical concerns that should be addressed before implementation.

## Strengths

1. **Excellent research quality.** The detection research document covers all major vectors (metadata, visual classifiers, behavioral signals, C2PA, text NLP, community reports) with specific technical detail and credible sources. The ranked priority table in Section 9 is particularly useful.

2. **Correct pipeline mapping.** The plan's Section 2 ("Current Pipeline Architecture") accurately maps the full pipeline flow: `generate_pin_content.py` -> `pin_assembler.py` -> `publish_content_queue.py` -> `blog_deployer.py` -> `post_pins.py`. The insertion points are correctly identified.

3. **Accurate function and file references.** The plan correctly references:
   - `PinAssembler._optimize_image()` (line 548 in `pin_assembler.py`)
   - `render_pin()` (line 441) and `render_batch()` (line 599)
   - `_deploy_blog_posts()` in `blog_deployer.py` (line 570)
   - Jitter constants in `post_pins.py` (lines 66-68)
   - The JPEG conversion logic in `_optimize_image()` (lines 571-592)
   - Prompt files `prompts/pin_copy.md` and `prompts/image_prompt.md`

4. **Correct identification of existing strengths.** The plan correctly notes that text overlay compositing, Puppeteer re-rendering, JPEG conversion for large images, jitter in posting, and description variety already provide partial protection.

5. **Sound technical approach.** The Pillow-based metadata stripping via re-save (not copying metadata to new image) is the correct approach for a Python-only solution. Avoiding ExifTool as a system dependency is a good call for GitHub Actions portability.

## Gaps and Concerns

### 1. Missed pipeline entry point: `publish_content_queue.py` uploads images to GCS/Drive

The plan does not mention `src/publish_content_queue.py`. This script uploads pin images and blog hero images to Google Cloud Storage (or Google Drive as fallback) for inline preview in the Google Sheets Content Queue. These uploaded images are the same files generated by `pin_assembler.py`.

**Impact:** If image cleaning is applied after `_optimize_image()` in `pin_assembler.py` (as the plan proposes), the cleaned images are the ones that `publish_content_queue.py` will upload, so this is actually fine. However, `publish_content_queue.py` also uploads hero images from `DATA_DIR / "generated" / "blog"`, which are the raw AI-generated images (not the rendered pins). These hero images flow to GCS/Drive and could retain AI metadata.

**Recommendation:** The plan should explicitly note that `publish_content_queue.py` uploads are covered because cleaning happens earlier in the pipeline. The hero image cleaning proposed for `blog_deployer.py` (Phase 1C) should also be applied earlier, in `generate_pin_content.py` at the point where the hero image is saved, so that ALL downstream consumers (publish_content_queue, blog_deployer, and the rendered pin itself) receive cleaned hero images.

### 2. Pin image format change (PNG to JPEG) may break downstream path references

The plan proposes that `clean_image()` converts PNG to JPEG. The current `_optimize_image()` already handles this case carefully: when converting to JPEG, it renames the `.jpg` file back to the original `.png` path (lines 584-585 in `pin_assembler.py`) so all downstream path references remain valid.

The plan's `clean_image()` returns a new path with `.jpg` extension and then the plan says:
```python
if cleaned_path != output_path:
    if output_path.exists():
        output_path.unlink()
    output_path = cleaned_path
```

**Problem:** This changes the path from `.png` to `.jpg`, but the `output_path` variable in `render_pin()` is local. The caller (`generate_pin_content.py` at line 165-171) expects the pin to be at `PIN_OUTPUT_DIR / f"{pin_id}.png"`. If `render_pin()` returns a `.jpg` path, the `image_path` stored in pin_data (line 197) would have the `.jpg` extension, which should propagate correctly. However, this needs careful testing because:

- `publish_content_queue.py` looks for pin images by pin_id in the output directory
- GCS/Drive upload functions may glob for files by pin_id
- The pin schedule stores `image_path` and `post_pins.py` reads it

**Recommendation:** Follow the existing pattern in `_optimize_image()`: rename the cleaned `.jpg` back to the original `.png` path. This avoids breaking any downstream path assumptions. The file content will be JPEG regardless of extension, and this is what the existing code already does.

### 3. Plan says to remove JPEG conversion from `_optimize_image()` — this is premature

The plan's Phase 1B, item 4 says: "In `_optimize_image()`, remove the JPEG conversion logic (lines ~571-592) since `clean_image()` now handles the JPEG conversion and does it better."

**Concern:** `_optimize_image()` serves a different purpose (file size reduction for the 500KB PNG target) than `clean_image()` (metadata stripping + anti-detection). Removing size optimization from `_optimize_image()` means the pipeline would always produce large PNGs first, then clean_image() would convert them to JPEG. This is functionally equivalent but changes the existing behavior in a way that conflates two concerns.

**Recommendation:** Keep `_optimize_image()` intact. Have `clean_image()` run after it. The `clean_image()` function already handles both PNG and JPEG input, so it will work regardless of whether `_optimize_image()` already converted to JPEG.

### 4. Jitter constant discrepancy in existing code

The plan says the current `INITIAL_JITTER_MAX = 900` (0-15 minutes) and proposes changing it to 5400 (0-90 minutes). However, the `apply_jitter()` docstring at `post_pins.py:300` already says `random(0, 5400)`, contradicting the actual constant value of 900 at line 66.

**Observation:** The docstring appears to have been updated ahead of the constant, or was written aspirationally. The plan should note this discrepancy and fix both the constant and the docstring together. Currently the docstring is misleading.

### 5. Workflow timeout increase is critical and undersized

The plan proposes increasing workflow timeouts from 30 to 120 minutes. With an initial jitter of up to 90 minutes plus inter-pin jitter of up to 40 minutes for the evening slot (2 pins), the worst case is 90 + 40 = 130 minutes. A timeout of 120 minutes would be insufficient in worst cases.

**Also note:** The evening workflow (`daily-post-evening.yml`) already has `timeout-minutes: 45`, not 30. The plan incorrectly states all three are at 30 minutes.

**Recommendation:** Set timeouts to 150 minutes for all three workflows. This accommodates 90 min initial jitter + 40 min inter-pin jitter + ~20 min for actual pin posting and network operations.

### 6. `blog_deployer.py` integration point is less precise than it could be

The plan says to clean hero images "in `_deploy_blog_posts()`, before building posts_for_commit." However, looking at the actual code at lines 599-633, hero images are located at lines 602-613 using a search loop across multiple candidate paths and extensions. The `clean_image()` call would need to:

1. Run after the hero image path is resolved (after line 613)
2. Handle the case where `hero_image_path` is `None` (line 615)
3. Update `hero_image_path` to the new (possibly different-extension) path

The plan's pseudocode is too simplified:
```python
if hero_image_path:
    hero_image_path = clean_image(hero_image_path)
```

This is correct in spirit but the reviewer notes that this must be placed after line 613 and before line 627 where posts_for_commit is built. The returned path from `clean_image()` may have a `.jpg` extension even if the input was `.png`, which would affect the `actual_ext` calculation at line 620 and the frontmatter regex replacement at lines 621-625.

**Recommendation:** Clean the hero image immediately after it is resolved (after line 613, before line 619). Update `hero_image_path` to the returned path. This ensures `actual_ext` picks up the correct extension.

### 7. `deploy_approved_posts()` is another entry point the plan does not cover

`blog_deployer.py` has a second deployment method: `deploy_approved_posts(posts_dir)` at line 467. This is an alternative entry point that reads MDX and hero images from a directory and commits them. It also has a hero image search loop (lines 498-504) that should receive the same `clean_image()` treatment.

**Recommendation:** Apply `clean_image()` in `deploy_approved_posts()` at line 507, before the post is added to `posts_to_deploy`.

### 8. No mention of the `generate-content.yml` workflow

The plan lists workflow changes for the three daily-post workflows but does not mention `.github/workflows/generate-content.yml`. Since the image cleaning is integrated at the Python module level (in `pin_assembler.py`), no workflow change is needed. However, the plan should explicitly note that `generate-content.yml` does not need modification and explain why, for completeness.

### 9. `numpy` dependency — consider pinning more tightly

The plan proposes `numpy>=1.24.0`. On GitHub Actions with Python 3.11, this could pull in numpy 2.x which had breaking changes. Since the usage is simple (array operations), numpy 2.x should work fine, but pinning to `numpy>=1.24.0,<3.0` would be safer.

## Recommendations

1. **Phase 1 is high-value, low-risk.** Metadata stripping via Pillow re-save is the single most impactful change. It should be implemented first and deployed immediately, before the other phases.

2. **Test with `piexif` or `Pillow.getexif()` to verify cleaning.** The testing strategy mentions verifying "zero metadata fields" but should specify the tool. Use `Pillow`'s `Image.open(cleaned).getexif()` for EXIF, and `Image.open(cleaned).info` for PNG text chunks. Also verify the JPEG file with `exiftool -all cleaned.jpg` manually before the first production run.

3. **Consider cleaning hero images at the point of generation** (in `_source_ai_image()` in `generate_pin_content.py`, around line 738) rather than only at deployment. This ensures ALL downstream consumers (publish_content_queue, blog_deployer, the pin assembler's data URI encoding) work with clean images from the start. This is a better architectural choice than cleaning at multiple downstream points.

4. **The text humanization additions (Phase 3) are low-risk and could be done in parallel with Phase 1.** Since they only modify prompt files, they cannot break anything and will improve copy quality regardless of AI detection.

5. **Add monitoring for the "Gen AI" label.** After implementation, manually check posted pins for the label. Consider adding a field to content-log.jsonl to track which pins receive the AI label, enabling data-driven evaluation of the avoidance measures' effectiveness.

6. **Pillow's `Image.open()` does NOT strip metadata by default** when you just open and resave. Specifically, `img.save("out.jpg")` on a JPEG input will copy EXIF data if present. The plan's `clean_image()` is correct in that it opens the image and saves without passing `exif=` argument, which for JPEG means no EXIF is written. However, for PNG -> JPEG conversion, PNG text chunks are naturally not carried over. The implementation should add an explicit test for this: open a JPEG with EXIF, resave, verify EXIF is absent.

   **Important nuance:** Pillow's behavior differs by version. In Pillow >= 10.0, `Image.save("JPEG")` without `exif=` does NOT copy EXIF. In older versions it might. Since the project pins `Pillow>=10.0.0`, this is fine, but the test should verify this explicitly.

## Summary

The research and plan are solid. The main findings that need correction before implementation:

- **Do not remove JPEG conversion from `_optimize_image()`** — keep existing behavior, add cleaning as a new step after it
- **Fix workflow timeout values** (evening is 45, not 30; target should be 150, not 120)
- **Handle format change propagation** by renaming .jpg back to .png path (matching existing pattern)
- **Cover `deploy_approved_posts()`** as a second entry point in blog_deployer.py
- **Consider cleaning at generation time** rather than at multiple deployment points

None of these are blockers. They are corrections that can be addressed during implementation.
